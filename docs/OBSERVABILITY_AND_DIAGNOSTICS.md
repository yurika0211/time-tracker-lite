# 观测与诊断设计

## 1. 目标

- 让问题可定位。
- 让异常可复现。
- 让损坏快照和迁移失败有清晰证据。

## 2. 观测对象

重点记录这些事件：

- 数据库打开。
- schema 校验。
- 计时开始、暂停、继续、停止。
- 记录新增、编辑、删除。
- 导出开始和结束。
- 清空和恢复。
- 迁移失败或归一化失败。

## 3. 日志原则

- 日志要短。
- 日志要结构化。
- 日志要避免泄露敏感文本。
- 日志要能回答“发生了什么、在哪一步、影响什么”。

## 4. 诊断信息

建议页面提供：

- 当前应用版本。
- 当前 schema 版本。
- 记录数量。
- 项目数量。
- 是否存在 active timer。
- 最近一次保存是否成功。
- 最近一次恢复是否成功。

## 5. 错误呈现

- 用户提示要直接。
- 技术错误和用户提示分开。
- 不要把完整堆栈直接暴露给普通界面。
- 必要时提供“复制诊断信息”入口。

## 6. 异常分类

- 存储不可用。
- schema 不一致。
- 数据归一化失败。
- 导出失败。
- 恢复失败。
- 未来同步冲突。

## 7. 调试建议

建议保留这些调试入口：

- 导出诊断快照。
- 查看当前统计摘要。
- 检查本地快照大小。
- 触发一次恢复校验。

## 8. 结论

这个项目不需要复杂监控平台，但必须有足够的本地诊断信息。否则一旦快照坏掉，排查成本会很高。
