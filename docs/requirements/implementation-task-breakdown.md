# 实现任务拆解

## 1. 目标

- 把需求文档变成可执行开发任务
- 每个任务都能单独开发、单独验收、单独回滚
- 先完成 P0 闭环，再做 P1 扩展，最后保留 P2 接口

## 2. 当前实现基线

- `src/main.ts` 已覆盖开始、暂停、继续、停止、项目添加、搜索、项目筛选、删除、CSV 导出、清空、空格键、刷新恢复
- `src/db.ts` 已用 sql.js 把 SQLite 快照落到 localStorage
- `src/types.ts` 只有最小状态定义
- 当前缺口是业务表、客户/标签、手动补录、编辑、报表、同步预留和测试

## 3. 拆分原则

- 一次只改一层
- 先数据，再逻辑，再界面
- schema 变更必须带迁移
- UI 变更必须同步状态与渲染
- 每个任务必须有明确完成标准

## 4. 任务序列

### 阶段 A：数据底座

- 任务 A1：拆分领域类型
  - 交付：`Project`、`Client`、`Tag`、`TimeEntry`、`ActiveTimer`、`Filters`、`Settings`
  - 完成标准：业务实体不再混在一个状态对象里
- 任务 A2：升级 SQLite 结构
  - 交付：`clients`、`projects`、`tags`、`time_entries`、`time_entry_tags`、`active_timer`、`settings`、`migrations`
  - 完成标准：新数据直接进表，旧快照可迁移
- 任务 A3：封装存储适配器
  - 交付：打开、读取、写入、持久化、清空、恢复接口
  - 完成标准：页面层不直接接触存储细节
- 任务 A4：旧数据归一化
  - 交付：名称清洗、时间戳修复、标签去重、缺失值补全
  - 完成标准：脏数据不会阻断启动

### 阶段 B：核心计时闭环

- 任务 B1：抽出计时状态机
  - 交付：开始、暂停、继续、停止的纯逻辑
  - 完成标准：同一时刻只允许一个 active timer
- 任务 B2：补齐恢复逻辑
  - 交付：刷新后恢复运行态或暂停态
  - 完成标准：页面重载后状态一致
- 任务 B3：计时展示与统计
  - 交付：elapsed、today、week、running count、entry count
  - 完成标准：统计基于时间戳计算，不依赖轮询次数
- 任务 B4：键盘快捷键
  - 交付：空格键切换计时
  - 完成标准：输入框内不触发，全局区域可触发

### 阶段 C：记录管理

- 任务 C1：记录列表数据源切换到业务表
  - 交付：entries 读写 SQLite 表
  - 完成标准：删除、刷新后数据保持一致
- 任务 C2：搜索与筛选
  - 交付：任务名、项目、标签关键词过滤
  - 完成标准：项目筛选与关键词筛选可叠加
- 任务 C3：删除与空态
  - 交付：单条删除、无结果空态
  - 完成标准：删除只影响目标行
- 任务 C4：CSV 导出
  - 交付：完整字段导出
  - 完成标准：表格软件可直接打开

### 阶段 D：项目、客户、标签

- 任务 D1：项目管理
  - 交付：新增、改名、归档、颜色、默认计费
  - 完成标准：重名受控，归档不影响历史记录
- 任务 D2：客户管理
  - 交付：客户增删改、项目绑定
  - 完成标准：项目可正确归属客户
- 任务 D3：标签管理
  - 交付：标签复用、去重、重命名、合并
  - 完成标准：标签输入不会产生重复脏值

### 阶段 E：手动补录与编辑

- 任务 E1：新增时间记录
  - 交付：独立录入入口
  - 完成标准：可绕过运行计时直接新增记录
- 任务 E2：编辑时间范围和元数据
  - 交付：任务名、项目、标签、计费、开始时间、结束时间可改
  - 完成标准：编辑后时长自动重算
- 任务 E3：重叠校验
  - 交付：时间区间合法性检查
  - 完成标准：非法区间不能保存
- 任务 E4：复制记录
  - 交付：从历史记录快速生成新记录
  - 完成标准：复制后保持字段完整

### 阶段 F：报表与汇总

- 任务 F1：日、周、月统计
  - 交付：固定时间窗口汇总
  - 完成标准：统计口径和日期边界一致
- 任务 F2：按项目、客户、标签汇总
  - 交付：多维分组汇总视图
  - 完成标准：同一条记录可被正确归类
- 任务 F3：自定义时间范围
  - 交付：起止日期筛选
  - 完成标准：范围外数据不参与统计
- 任务 F4：汇总导出
  - 交付：可导出统计结果
  - 完成标准：结果可直接复用到表格软件

### 阶段 G：同步预留

- 任务 G1：同步接口抽象
  - 交付：本地数据与同步数据的隔离接口
  - 完成标准：本地模式不依赖网络
- 任务 G2：冲突处理策略
  - 交付：覆盖、合并、保留本地的处理规则
  - 完成标准：同步不会静默丢数据
- 任务 G3：登录和账号入口预留
  - 交付：仅保留入口和状态位
  - 完成标准：默认关闭，不影响单机使用

## 5. 推荐执行顺序

- 先做阶段 A，再做阶段 B
- 然后做阶段 C，补齐当前记录闭环
- 再做阶段 D 和 E
- 最后做阶段 F 和 G

## 6. 任务粒度

- 单个任务尽量控制在 2 到 6 小时
- 每个任务只改一个主层次
- 任何 schema 变更都要带迁移和文档更新
- 任何 UI 变更都要补状态和渲染检查

## 7. 暂不进入 P0

- 登录
- 云同步
- 团队共享
- 自动追踪
- 空闲提醒
- 日历视图
- PDF 导出
