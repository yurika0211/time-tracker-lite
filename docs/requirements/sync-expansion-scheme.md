# 同步扩展方案

## 1. 目标

- 保持本地优先
- 未来需要多设备时再接入同步
- 不改动现有单机体验
- 不让同步反过来限制单机可用性

## 2. 设计原则

- 本地 SQLite 是主事实来源
- 同步层只负责传输、记录和冲突处理
- 所有写入先落本地，再进入同步队列
- 所有同步操作都要可重试、可回放
- 任何冲突都不能静默覆盖

## 3. 同步边界

### 3.1 建议同步的对象

- `clients`
- `projects`
- `tags`
- `time_entries`
- `time_entry_tags`
- 与账号相关的部分 `settings`

### 3.2 不同步的对象

- `active_timer` 的运行态
- `filters`
- 临时 UI 状态
- 导出文件
- 缓存快照

### 3.3 处理方式

- `active_timer` 只在本地运行
- 停止后转成 `time_entries` 再同步
- 只读状态可以同步，临时状态不进远端

## 4. 数据预留

### 4.1 业务表建议增加的字段

- `id`：全局唯一标识
- `device_id`：来源设备
- `version`：本地版本号
- `updated_at`：最后更新时间
- `deleted_at`：删除时间，空值表示未删除

### 4.2 本地同步辅助表

#### `sync_outbox`

- `op_id`
- `entity_type`
- `entity_id`
- `op_type`
- `payload`
- `created_at`
- `retry_count`
- `status`
- `last_error`

#### `sync_cursor`

- `target`
- `cursor`
- `updated_at`

#### `sync_conflicts`

- `conflict_id`
- `entity_type`
- `entity_id`
- `local_snapshot`
- `remote_snapshot`
- `reason`
- `created_at`
- `resolved_at`

## 5. 同步流程

1. 本地修改先写入主表
2. 同步操作写入 `sync_outbox`
3. 发起 push，把待同步变更送到服务端
4. 发起 pull，拉取远端增量
5. 按规则应用远端变更
6. 推进游标
7. 有冲突时写入冲突表，不直接覆盖

## 6. 冲突处理规则

- 同一条记录的不同字段被修改：允许合并
- 同一字段被多设备修改：进入冲突，不自动覆盖
- 删除和修改同时出现：以更晚的有效操作为准；无法判断时进入冲突
- 同一 `id` 的重复创建：理论上应避免，出现时进入冲突
- 时间条目如果因同步产生重叠：不自动改写，交给人工处理
- `active_timer` 不做跨设备自动合并

## 7. API 预留

- `GET /sync/bootstrap`
- `POST /sync/push`
- `GET /sync/pull?cursor=...`
- `POST /sync/resolve`

## 8. 分阶段落地

### 阶段 1：离线导入导出

- 支持快照导入 / 导出
- 先验证数据结构稳定性

### 阶段 2：单用户多设备同步

- 同一账号在多个设备间同步
- 先保证数据不丢

### 阶段 3：冲突处理界面

- 展示冲突记录
- 允许用户选择保留本地、保留远端或手动合并

### 阶段 4：账号与鉴权

- 补登录态
- 补访问控制

### 阶段 5：团队共享

- 仅作为后续可能，不进当前 MVP

## 9. 验收标准

- 离线修改可以排队
- 重新联网后可以同步
- 断网重试不会丢数据
- 出现冲突时有明确记录
- 同步后本地页面仍可继续使用
- 远端异常不会破坏本地数据

## 10. 暂不做

- 实时协作
- 在线同时编辑同一条记录
- 自动解决所有冲突
- 公开分享链接
- 以同步系统取代本地存储
