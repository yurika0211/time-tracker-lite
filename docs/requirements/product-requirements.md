# 需求分析：Time Tracker Lite

## 1. 产品定位

- 本地优先的轻量计时工具
- 替代散装记录、手写表格和浏览器临时存储
- 先做个人单机闭环，再考虑同步和协作

## 2. 需求原则

- 业务数据只保留一份事实来源
- 计时、统计、导出都必须可重放
- 展示层和业务层分开
- 先保证准确，再谈美观
- 先保证可恢复，再谈扩展

## 3. 版本分层

### P0：必须完成

- 开始、暂停、继续、停止计时
- 任务名称、项目、标签、计费标记
- 记录列表、搜索、项目筛选、删除
- 今日、本周、进行中、条目总数统计
- 新增项目
- CSV 导出
- 清空数据
- 刷新恢复
- 空格键快捷操作

### P1：尽快补齐

- 手动新增记录
- 编辑已保存记录
- 客户管理
- 日期范围筛选
- 复制记录
- 项目归档与项目颜色
- 标签管理页
- 更完整的汇总报表

### P2：后续扩展

- 登录和账号体系
- 多设备同步
- 团队共享
- 自动追踪
- 空闲提醒
- 日历视图
- 发票、预算、审批

## 4. 功能需求

### 4.1 核心计时

- 页面只允许一个进行中的计时
- 未开始时可填写任务名称、项目、标签、计费标记
- 开始后生成当前 active timer
- 已有 active timer 时，不允许再新建第二条
- 运行中修改表单，必须同步到当前 active timer
- 暂停后停止累计当前片段
- 继续后从新的片段开始累计
- 停止后生成一条历史记录并清空 active timer
- 计时结果必须基于时间戳计算，不依赖轮询次数
- 页面刷新后，运行态或暂停态都应恢复
- 输入为空时，任务名称默认使用“未命名任务”
- 在非输入框区域按空格键，应切换开始、暂停、继续状态

### 4.2 记录管理

- 历史记录按开始时间倒序展示
- 搜索范围包括任务名称、项目名、标签
- 项目筛选与关键词搜索可以同时生效
- 无结果时显示空态
- 删除只影响单条记录，不影响其他记录
- 常用项目区域按累计时长排序，默认展示前 8 个
- 过滤条件应在刷新后保留

### 4.3 项目、客户、标签、计费

- 项目是主要分组维度
- 客户是项目的上级维度，一个项目最多对应一个客户
- 标签支持多值输入，逗号分隔
- 标签需要去空格、过滤空值、避免重复
- 计费标记以记录为准，项目可提供默认值
- 项目可归档，归档后默认不出现在快捷选择中，但历史记录仍可查看
- 项目可配置颜色，用于列表和统计识别
- 新增项目时必须去重，名称不能只含空白
- 默认项目集合可以预置，但必须允许后续调整
- 历史记录保存项目和客户的标识，名称变更不应破坏历史数据

### 4.4 手动补录与编辑

- 允许不经过运行计时直接新增记录
- 允许编辑已保存记录的名称、项目、标签、计费标记、开始时间和结束时间
- 编辑时间范围后，时长必须重新计算
- 开始时间必须早于结束时间
- 不能制造重叠记录；如用户操作会产生重叠，必须拦截并提示
- 允许把现有记录复制成新记录

### 4.5 报表、统计与导出

- 今日统计按本地日期 00:00 到当前时刻计算
- 本周统计按周一 00:00 开始计算
- 进行中的计时必须计入统计
- 支持按项目、标签、客户汇总
- 支持自定义时间范围统计
- CSV 导出必须包含任务名称、项目、客户、标签、计费标记、开始时间、结束时间、时长
- 导出文件应兼容表格软件直接打开
- 默认导出全量数据，后续可再补当前筛选导出

### 4.6 登录、同步和多设备

- MVP 不做登录和云同步
- 产品必须离线可用
- 后续若加入同步，本地 SQLite 仍必须是主事实来源
- 同步必须有冲突处理策略，不能靠覆盖了事
- 多设备一致性是后续能力，不进入当前最小版本

## 5. 数据规则

- 时间字段统一使用毫秒时间戳
- durationMs 必须为非负整数
- duration 应由开始和结束时间推导，不允许手填成事实
- active timer 同时只能有一条
- project、client、tag 名称必须 trim 后再保存
- tag 名称空值无效
- 缺失字段和旧数据在加载时要做归一化
- 派生统计不落盘
- 设置项和业务数据分开存
- UI 状态可以缓存，但不能代替业务事实

## 6. 非功能要求

- 本地优先，MVP 不依赖网络
- 关键渲染和输入必须避免不受控的 HTML 注入
- 面向现代桌面浏览器
- 所有核心操作都要可键盘完成
- 核心逻辑必须有测试覆盖
- 提交前必须通过类型检查和构建
- 常规数据量下，列表和统计不能明显卡住界面
- 迁移存储结构时必须保留兼容说明
- 错误提示要直接、可执行

## 7. 待确认问题

- CSV 默认导出全量还是当前筛选结果
- 手动编辑放在独立页面还是弹窗
- 客户是否必须绑定到项目
- 标签大小写是否视为不同值
- 项目颜色是固定色板还是自定义值
- 计费默认值来自项目还是只认记录
- 允许的最大标签数量

## 8. 实现约束摘要

- 详细架构约束见 `../ARCHITECTURE_CONSTRAINTS.md`
- 业务层、应用层、渲染层、存储层必须分开
- SQLite 是唯一事实来源
- `localStorage` 只能承载快照或辅助状态
- 核心逻辑优先写成纯函数
- 验收标准见 `./testing-acceptance-standards.md`
- 同步扩展方案见 `./sync-expansion-scheme.md`
- 具体任务拆解见 `./implementation-task-breakdown.md`
