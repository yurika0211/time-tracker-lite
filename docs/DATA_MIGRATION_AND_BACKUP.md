# 数据迁移与备份方案

## 1. 现状

当前项目采用 `sql.js` 在浏览器里维护 SQLite，再把二进制快照写入 `localStorage`。

相关键值：

- `ttl.sqlite`：SQLite 二进制快照，Base64 编码后保存。
- `ttl.schema`：当前 schema 标记。

这个方案的优点是简单、离线可用、迁移成本低；缺点是快照可被篡改、容量有限、恢复必须做得更稳。

## 2. 迁移目标

- 旧数据可以正常打开。
- 新字段可以平滑加入。
- schema 失败时可回退。
- 迁移过程不破坏用户已有记录。
- 派生数据始终可重新计算。

## 3. 版本管理

建议把版本管理拆成两层：

- 数据库版本：表结构变化的版本号。
- 应用版本：前端代码版本号。

每次 schema 变化都要满足：

- 迁移脚本可重复执行。
- 迁移前后数据数量和关键字段一致。
- 迁移说明和代码同时更新。

## 4. 迁移流程

1. 打开快照。
2. 读取当前 schema 版本。
3. 按顺序执行增量迁移。
4. 校验核心不变量。
5. 写回新的快照。
6. 若失败，回退到上一个稳定快照或重建空库。

## 5. 迁移原则

- 只做增量，不随意重写历史事实。
- 迁移脚本要幂等。
- 迁移失败不能让页面静默崩溃。
- 迁移过程要保留足够的诊断信息。
- 新字段先默认兼容旧数据，再逐步收紧约束。

## 6. 备份策略

### 6.1 轻量备份

- 在执行清空、重大编辑、批量导入前，先生成当前快照。
- 导出 CSV 只能算辅助备份，不能替代 SQLite 快照。

### 6.2 恢复入口

- 从历史快照恢复。
- 从文件导入恢复。
- 从当前浏览器存储恢复。

### 6.3 保留策略

- 至少保留最后一个稳定快照。
- 若后续接入自动备份，可保留最近若干份。
- 过期快照要有清理规则。

## 7. 损坏恢复

当快照损坏或 schema 不一致时，优先按以下顺序处理：

- 尝试读取并归一化。
- 读取失败则回滚到上一份快照。
- 再失败则提示导出失败原因并让用户决定是否清空重建。

恢复时必须明确告知：

- 哪些数据可以恢复。
- 哪些数据已经丢失或无法验证。
- 下一步该做什么。

## 8. 导入导出格式

- SQLite 快照是主备份格式。
- CSV 是辅助交换格式，只适合表格查看和轻量迁移。
- 如果未来增加 JSON 导入导出，必须先定义严格 schema。

## 9. 回滚规则

- 代码回滚：恢复到上一个稳定提交。
- 数据回滚：恢复到上一份可用快照。
- schema 回滚：只有在迁移脚本支持反向变更时才允许。

## 10. 结论

这个项目最怕的不是“没有新功能”，而是“老数据打不开”。所以迁移和备份必须比新功能更早做稳。
