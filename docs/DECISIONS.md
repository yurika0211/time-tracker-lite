# 设计决策记录

## 1. TypeScript

- 决策：前端与业务逻辑统一用 TypeScript。
- 原因：类型约束更适合拆分状态、校验和迁移。
- 状态：已确定。

## 2. npm

- 决策：统一使用 npm 管理依赖。
- 原因：安装简单，和 Vite 配合直接。
- 状态：已确定。

## 3. SQLite

- 决策：SQLite 作为单一事实来源。
- 原因：适合本地优先、离线可用、结构化存储。
- 状态：已确定。

## 4. sql.js

- 决策：当前浏览器端用 sql.js 作为 SQLite 运行时。
- 原因：便于先做本地 MVP。
- 状态：已确定，但后续可替换实现。

## 5. 目录分层

- 决策：UI、应用、领域、基础设施分层。
- 原因：减少耦合，方便测试。
- 状态：已确定。

## 6. 同步

- 决策：同步不进 MVP。
- 原因：先把单机闭环做稳。
- 状态：暂缓。

## 7. 导出

- 决策：先做 CSV，不优先做 PDF。
- 原因：表格兼容性最好，成本低。
- 状态：已确定。

## 8. 备份

- 决策：SQLite 快照优先，CSV 只作辅助。
- 原因：业务事实需要可完整恢复。
- 状态：已确定。

## 9. 后续待定

- 迁移到真实后端时是否继续保留 sql.js。
- 是否引入独立报表页。
- 是否引入分页或虚拟列表。
- 是否增加导入向导。
