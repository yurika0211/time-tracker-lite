# 开发文档：架构约束

## 目的

- 让 Time Tracker Lite 按企业级方式演进，而不是停留在单文件原型
- 防止业务逻辑、渲染、存储混在一起
- 保证后续可以平滑接入真正后端、同步和多端能力

## 架构模型

- **UI 层**：只负责 DOM、事件绑定、渲染
- **应用层**：只负责用例编排，例如开始、暂停、停止、导出、筛选
- **领域层**：只负责时间计算、记录规则、数据校验、状态机
- **基础设施层**：只负责 sql.js、SQLite 快照、CSV 导出、时间源适配

## 强制约束

- **单一事实来源**：业务数据只认 SQLite；`localStorage` 只能保存 SQLite 二进制快照，不存业务字段
- **单向依赖**：UI → 应用 → 领域 → 基础设施，禁止反向引用
- **纯逻辑下沉**：时长计算、标签解析、过滤、汇总必须是纯函数
- **显式状态**：`active timer`、`entries`、`projects`、`filters` 必须全部显式建模
- **无隐式全局**：禁止把运行态藏在闭包、DOM 属性或临时变量里
- **边界校验**：所有用户输入在进入领域前完成 `trim`、去空、类型校验
- **写入原子**：任何持久化变更必须先更新内存状态，再一次性写回
- **派生数据不落盘**：统计值、展示文案、过滤结果不能当作事实数据保存
- **安全渲染**：用户输入只允许经过转义后进入 DOM；默认不使用未受控 `innerHTML`
- **统一时间源**：业务函数接收时间参数；`Date.now()` 只能出现在入口层、定时器或时钟适配器
- **失败可恢复**：数据库损坏、字段缺失、旧版本数据必须可归一化后继续运行
- **变更可回滚**：每次 schema 或存储格式变更都要写迁移说明

## 推荐目录

- `src/app/`：启动装配
- `src/domain/`：纯规则
- `src/application/`：用例编排
- `src/infrastructure/`：SQLite、存储、导出
- `src/ui/`：DOM 与渲染
- `src/shared/`：通用工具

## 质量门禁

- 核心逻辑必须有测试：时长计算、暂停/继续、过滤、导出、序列化
- 提交前必须通过 `npm run typecheck` 和 `npm run build`
- 任何新增依赖都要说明原因和替代方案
- 任何影响存储的改动都要检查导出兼容性
- 性能回归必须在开发阶段拦截，不能留到上线后再补

## 非目标

- 不在 MVP 里引入多服务拆分
- 不把数据库逻辑散落在页面代码里
- 不让 UI 直接访问数据库文件细节
- 不引入多个状态源来“临时兼容”

## 变更原则

- 先补文档，再改代码
- 先补约束，再扩功能
- 先保兼容，再做优化
- 任何涉及数据结构的改动，必须同步更新文档
