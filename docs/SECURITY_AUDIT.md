# 安全审计设计

## 1. 审计范围

当前版本是纯前端 + 本地 SQLite 快照方案，安全审计重点放在这些边界：

- 页面输入。
- DOM 渲染。
- CSV 导出。
- 本地持久化。
- sql.js 依赖。
- 未来同步和账号入口。

## 2. 资产

需要保护的对象包括：

- 时间记录。
- 项目、客户、标签。
- 运行中的 active timer。
- SQLite 快照文件。
- 未来可能加入的账号态、同步态、访问令牌。

## 3. 主要威胁面

### 3.1 DOM 注入

用户输入如果直接进入 HTML，可能形成脚本执行或界面篡改。

### 3.2 CSV 注入

导出的内容如果以 `=`, `+`, `-`, `@` 开头，可能被表格软件当公式处理。

### 3.3 本地存储篡改

`localStorage` 可被浏览器插件、调试工具或恶意脚本修改，数据不能直接当成可信输入。

### 3.4 快照损坏

SQLite 快照、schema 版本或历史数据可能因为旧版本、截断、脏写而失效。

### 3.5 供应链风险

`sql.js`、构建工具和未来新增依赖都可能引入安全问题。

### 3.6 未来同步风险

如果以后接入账号和同步，冲突覆盖、令牌泄漏、权限过宽都会变成高风险点。

## 4. 主要控制措施

- 所有用户文本默认走 `textContent` 或转义后再显示。
- 禁止未受控 `innerHTML`、`eval`、内联事件处理。
- CSV 导出时对首字符风险做转义处理。
- 进入领域层前先做 `trim`、去空、类型校验、长度限制。
- 加载数据时做归一化，不让脏数据直接阻塞启动。
- `localStorage` 只存快照，不存密钥和账号信息。
- 依赖版本固定，升级前先审计。
- 若未来上线，补严格的内容安全策略。
- 同步能力默认关闭，未设计好前不开放入口。

## 5. 审计清单

### 5.1 输入面

- 任务名、项目名、客户名、标签是否都做了去空和长度限制。
- 是否有未校验的 JSON 反序列化入口。
- 是否允许超长字符串把页面拖慢。

### 5.2 渲染面

- 任何用户文本是否都经过安全渲染。
- 是否存在把拼接字符串直接塞进 HTML 的地方。
- 是否有第三方组件插入未净化内容。

### 5.3 导出面

- CSV 是否处理了公式注入。
- 是否处理了逗号、换行、引号和非 ASCII 字符。
- 导出文件名是否避免敏感信息泄漏。

### 5.4 存储面

- 快照是否校验 schema 版本。
- 损坏数据是否会被安全降级。
- 清空和恢复是否需要明确确认。

### 5.5 依赖面

- 是否锁定版本。
- 是否在升级前跑类型检查、构建和安全检查。
- 是否记录关键依赖变更理由。

### 5.6 未来同步面

- 令牌是否只放在受控存储。
- 同步冲突是否有人工决策入口。
- 服务端返回是否默认不信任。

## 6. 风险等级建议

- 高：XSS、CSV 注入、令牌泄漏、静默覆盖远端数据。
- 中：快照损坏、依赖漏洞、同步冲突。
- 低：错误提示过长、调试信息泄漏、非关键字段格式异常。

## 7. 发布门禁

发布前至少确认以下事项：

- `npm run typecheck` 通过。
- `npm run build` 通过。
- 核心输入、渲染、导出路径检查完成。
- 快照恢复与异常数据恢复检查完成。
- 依赖版本没有已知高危漏洞。
- 文档中的安全约束已同步到实现。

## 8. 结论

当前项目的安全重点不是“网络攻击”，而是“本地输入、导出、快照和未来同步”这四个边界。先把这四个口子管住，再谈扩展。
