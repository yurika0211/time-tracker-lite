# 发布与回滚方案

## 1. 目标

- 发布前先锁风险。
- 出问题时能快速回到稳定版本。
- 保证数据和代码都能回退。

## 2. 发布前检查

- 类型检查通过。
- 构建通过。
- 核心流程手工验收通过。
- 刷新恢复通过。
- CSV 导出通过。
- 异常快照恢复通过。
- 文档索引已更新。

## 3. 发布内容

每次发布至少包含：

- 代码变更。
- 文档变更。
- 必要迁移说明。
- 已知问题列表。

## 4. 回滚层次

### 4.1 代码回滚

- 恢复到上一个稳定提交。
- 保留数据快照不动。

### 4.2 数据回滚

- 恢复上一份可用 SQLite 快照。
- 优先保留最近一次人工确认稳定的版本。

### 4.3 schema 回滚

- 只有在迁移脚本支持逆向时才做。
- 否则走前向修复，不直接硬退结构。

## 5. 风险动作

以下操作必须更谨慎：

- 清空数据。
- 覆盖快照。
- 大批量迁移。
- 改变导出字段。
- 修改计时状态机。

## 6. 发布节奏

建议按这个顺序：

1. 完成文档。
2. 完成代码。
3. 跑测试和 QA。
4. 备份快照。
5. 发布。
6. 观察首轮反馈。

## 7. 回滚触发条件

- 数据丢失。
- 统计错误。
- 刷新恢复失败。
- CSV 无法打开。
- schema 不兼容。
- 页面静默崩溃。

## 8. 结论

对这个项目来说，回滚不是补救手段，而是发布流程的一部分。没有回滚方案，就不算完成发布设计。
