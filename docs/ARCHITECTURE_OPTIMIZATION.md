# 架构优化设计

## 1. 目标

- 把当前单文件原型拆成可维护的分层结构。
- 让计时、存储、渲染、导出可以分别测试。
- 为后续同步、报表、权限和多端能力留出边界。
- 避免业务逻辑继续堆在 `src/main.ts` 里。

## 2. 现状基线

当前代码已经具备这些模块：

- `src/main.ts`：事件绑定、页面装配、状态协调。
- `src/db.ts`：sql.js 打开、写入、持久化、清空。
- `src/domain/state.ts`：状态归一化、标签解析、项目去重。
- `src/types.ts`：核心数据类型。

当前瓶颈也很明确：

- 业务规则和 UI 还没有完全拆开。
- 存储适配器和领域逻辑仍有耦合。
- 派生数据、展示状态、业务事实还需要再分层。
- 未来同步能力还没有明确接口边界。

## 3. 设计原则

- 单向依赖：UI → 应用 → 领域 → 基础设施。
- 领域纯函数：时长、状态切换、过滤、校验必须可独立运行。
- 显式状态：运行态、筛选态、草稿态分开建模。
- 时间源注入：业务逻辑不直接调用 `Date.now()`。
- 单一事实来源：SQLite 只保存业务事实。
- 派生数据不落盘：统计值只计算，不当真值存。
- 写入原子化：一次用例完成后再统一持久化。

## 4. 目标结构

建议目录继续向下拆成：

- `src/app/`：启动装配、依赖注入、页面入口。
- `src/application/`：用例编排，负责开始、暂停、继续、停止、导出等流程。
- `src/domain/`：状态机、校验、时间计算、归一化规则。
- `src/infrastructure/`：SQLite、快照、导出、未来同步适配器。
- `src/ui/`：DOM 查询、渲染、事件派发、可访问性处理。
- `src/shared/`：通用工具、格式化、错误码。

## 5. 核心优化项

### 5.1 状态模型拆分

建议把状态分成三层：

- 业务事实：项目、客户、标签、时间条目、运行计时器、设置。
- 会话状态：表单草稿、弹窗开关、筛选条件、确认状态。
- 派生状态：今日总时长、本周总时长、常用项目、筛选结果。

### 5.2 用例编排

每个用例都应是明确入口：

- 开始计时。
- 暂停计时。
- 继续计时。
- 停止并落盘。
- 新增或编辑项目。
- 新增或编辑时间记录。
- 导出 CSV。
- 恢复快照。

### 5.3 时间处理

- 业务层只接收时间参数，不自己取当前时间。
- `setInterval` 只负责刷新展示，不负责累计事实。
- 运行时长以时间戳差值计算。
- 暂停和继续要把片段边界显式建出来。

### 5.4 存储层

- 对外只暴露仓储接口，不暴露 sql.js 细节。
- 打开数据库时自动做 schema 校验和迁移。
- 写入时尽量走事务或一次性快照提交。
- `localStorage` 只保存 SQLite 二进制快照。

### 5.5 渲染层

- 默认只用 `textContent`，不用未受控 `innerHTML`。
- 渲染函数只消费状态，不直接改业务数据。
- 列表、统计、常用项目都应由状态推导。
- 常用项目和筛选结果可做缓存，但不能成为事实来源。

## 6. 迁移路径

1. 先拆类型，把业务实体从混合状态里分出来。
2. 再抽领域层，把计时、归一化、校验写成纯函数。
3. 再包存储层，把打开、保存、清空单独封装。
4. 再拆 UI 层，把渲染和事件处理分离。
5. 再补测试，先锁住纯逻辑和迁移逻辑。
6. 最后补同步预留和冲突处理入口。

## 7. 不做的事

- 不引入微服务拆分。
- 不让页面代码直接访问数据库细节。
- 不把同步系统做成主依赖。
- 不为了抽象而引入过重的状态框架。

## 8. 验收标准

- `main.ts` 里不再堆业务规则。
- 存储实现可替换，不影响应用层。
- 核心逻辑都有单元测试。
- 刷新恢复、导出、删除、筛选都可独立验证。
- 新增功能可以落在对应层，不需要改全局结构。
