# 测试与 QA 计划

## 1. 目标

- 把“能用”变成“可回归”。
- 给核心计时、存储、导出、恢复建立统一验证方式。
- 让后续新增功能能沿用同一套测试框架。

## 2. 测试分层

### 2.1 单元测试

覆盖纯逻辑：

- 状态机切换。
- 时间差计算。
- 标签拆分与去重。
- 项目去重与归一化。
- CSV 行生成。
- 数据迁移和归一化。

### 2.2 集成测试

覆盖模块协作：

- 打开、写入、恢复数据库。
- 开始、暂停、继续、停止。
- 刷新后恢复 active timer。
- 搜索、筛选、删除。
- 导出与列表一致性。

### 2.3 端到端检查

覆盖用户路径：

- 首次打开页面。
- 连续计时与刷新恢复。
- 新增项目并立即使用。
- 导出 CSV 并用表格软件打开。
- 损坏快照恢复。

## 3. 回归矩阵

建议每次发布前至少跑这些场景：

- 0 条记录。
- 1 条 active timer。
- 1000 条历史记录。
- 100 个项目。
- 200 个标签。
- 损坏快照样本。
- 清空后重建。

## 4. 验收门槛

- 类型检查通过。
- 构建通过。
- 核心逻辑测试通过。
- 刷新恢复通过。
- CSV 可打开。
- 异常数据不会让页面静默崩溃。

## 5. 手工检查清单

- 空格键是否只在非输入区触发。
- 搜索和项目筛选是否可叠加。
- 暂停后是否不再累计。
- 继续后是否从新片段计算。
- 删除是否只影响目标记录。
- 清空是否有二次确认。

## 6. 缺陷分级

- P0：数据丢失、统计错误、刷新恢复失败、导出损坏。
- P1：筛选异常、UI 不一致、快捷键误触。
- P2：文案、排版、边角交互问题。

## 7. QA 输出

每轮 QA 最少输出：

- 测试范围。
- 通过项。
- 失败项。
- 复现步骤。
- 修复建议。
- 是否允许发布。
